🎯 JERARQUÍA DE RESOLUCIÓN (CORE)
═══════════════════════════════════════════════════════════════════════════

Cuando se necesita saber qué turno tiene un empleado:

   ┌─────────────────────────────────────────────────────────────┐
   │ NIVEL 1: OVERRIDE INDIVIDUAL (máxima prioridad)             │
   │ ├─ ¿Existe employee_shift_overrides para hoy?              │
   │ └─ Sí → Retorna turno individual                           │
   └─────────────────────────────────────────────────────────────┘
                            ↓ No
   ┌─────────────────────────────────────────────────────────────┐
   │ NIVEL 2: TURNO GLOBAL POR PUESTO                            │
   │ ├─ ¿Existe global_schedules para puesto + día?             │
   │ └─ Sí → Retorna turno global                               │
   └─────────────────────────────────────────────────────────────┘
                            ↓ No
   ┌─────────────────────────────────────────────────────────────┐
   │ NIVEL 3: TURNO POR DEFECTO DE SUCURSAL                      │
   │ ├─ ¿Existe branch_default_shifts para sucursal + día?      │
   │ └─ Sí → Retorna turno sucursal                             │
   └─────────────────────────────────────────────────────────────┘
                            ↓ No
   ┌─────────────────────────────────────────────────────────────┐
   │ NIVEL 4: SIN ASIGNACIÓN                                     │
   │ └─ Retorna null (empleado sin turno asignado)              │
   └─────────────────────────────────────────────────────────────┘

✨ CARACTERÍSTICAS PRINCIPALES
═══════════════════════════════════════════════════════════════════════════

1. DRAG & DROP INTUITIVO
   ├─ Arrastra turnos de la librería a las celdas
   ├─ Visual feedback claro
   └─ Mobile-friendly (touch support)

2. OPTIMISTIC UPDATES
   ├─ UI responde al instante (sin esperar BD)
   ├─ Persistencia en background
   └─ Rollback automático si falla

3. BULK ACTIONS
   ├─ Ctrl+Click para seleccionar múltiples celdas
   ├─ Aplicar mismo turno a todas
   └─ Batch de 500 registros por request

4. JERARQUÍA CLARA
   ├─ Override > Global > Branch > None
   ├─ No hay ambigüedad
   └─ Fácil de entender y mantener

5. AUDITORÍA COMPLETA
   ├─ shift_change_logs registra todo
   ├─ Quién cambió, qué cambió, cuándo
   └─ Datos JSONB de old/new values

6. PERFORMANCE
   ├─ Caché en memoria (5 min)
   ├─ Índices en BD (O(log n))
   ├─ Soft deletes (sin cascadas)
   └─ Batching (500 reg/request)

🎓 CONCEPTOS CLAVE
═══════════════════════════════════════════════════════════════════════════

SHIFT TEMPLATE
└─ Definición reutilizable de un turno (Mañana, Tarde, Noche)
   Ejemplo: start_time: 08:00, end_time: 17:00, color: #3B82F6

GLOBAL SCHEDULE
└─ Asignación de un turno a un puesto en un día de semana
   Ejemplo: Caja (puesto) + Lunes → Turno Mañana

EMPLOYEE SHIFT OVERRIDE
└─ Sobreescribe la asignación global para un empleado en una fecha
   Ejemplo: Juan (empleado) + 2026-03-30 → Turno Tarde (en lugar de Mañana)

SHIFT RESOLUTION
└─ Proceso de determinar qué turno le corresponde a un empleado en una fecha
   Cascada: Override → Global → Branch → None

OPTIMISTIC UPDATE
└─ Actualizar UI sin esperar respuesta del servidor
   Ventaja: UX fluida
   Riesgo: Puede quedar inconsistente si BD rechaza

⚠️ PUNTOS CRÍTICOS A RECORDAR
═══════════════════════════════════════════════════════════════════════════

1. El UNIQUE constraint previene conflictos:
   UNIQUE(company_id, job_position_id, day_of_week)
   → Un puesto solo puede tener un turno por día

2. El resolver sigue la cascada estrictamente:
   Override (individual) > Global (puesto) > Branch (sucursal) > None
   → No hay ambigüedad

3. Soft deletes preservan auditoría:
   deleted_at !== NULL (no se elimina, se marca como eliminado)
   → Todos los queries filtran por deleted_at IS NULL

4. Caché tiene validez de 5 minutos:
   → Invalidar manualmente si necesario (clearCache())

5. Batch size es 500 registros:
   → Evita timeouts de API, optimiza performance
